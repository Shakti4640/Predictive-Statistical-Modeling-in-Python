Project 21: Random Forest for Credit Scoring — Ensemble Thinking

Bagging Random Forest Feature Importance Out-of-Bag Error Variance Reduction

Difficulty: Medium  |  Builds on: Projects 12, 15

1 · Hypothetical Problem

  • You work at a bank building a credit scoring model to predict loan defaults
  • Features: income, debt-to-income ratio, credit history length, number of open accounts, recent inquiries, employment years
  • A single decision tree (Project 12) overfits — different random seeds give wildly different trees
  • Pruning helps (max_depth from Project 12), but the model is still unstable
  • Goal → combine many trees into a forest that's more accurate and stable than any single tree

2 · How to Approach It

  1. Recall from Project 12: a single deep tree has high variance — small data changes produce very different trees
  2. Bagging (Bootstrap AGGregatING): train many trees on random subsets of data, average their predictions
  3. Random Forest adds another twist: at each split, only consider a random subset of features
  4. This double randomness (data + features) makes trees diverse — their errors cancel out when averaged
  5. Use cross-validation (Project 15) to compare single tree vs forest

3 · How Can We Solve It

  1. Generate credit scoring data with nonlinear patterns
  2. Train a single decision tree — show its instability across different seeds
  3. Train a Random Forest — show it's more stable and accurate
  4. Extract feature importances — which features drive credit decisions?
  5. Use out-of-bag (OOB) score as a free validation estimate
  6. Compare performance: single tree vs forest at different n_estimators

4 · What Should Do

  • ✅ Start with 100-500 trees — more trees rarely hurt (just slower)
  • ✅ Use oob_score=True for a free validation estimate without needing a separate validation set
  • ✅ Check feature_importances_ to understand what drives predictions
  • ✅ Compare with a single tree to demonstrate the ensemble benefit
  • ✅ Random Forest handles nonlinearity and interactions automatically — no polynomial features needed

5 · What Should NOT Do

  • ❌ Do NOT use a single tree when you can use a forest — almost always better
  • ❌ Do NOT scale features for Random Forest — tree-based models are scale-invariant (unlike Ridge/Lasso from Projects 17-18)
  • ❌ Do NOT use too few trees (n_estimators=5) — the variance reduction requires many trees
  • ❌ Do NOT interpret feature_importances_ as causal — it measures predictive power, not causation
  • ❌ Do NOT assume Random Forest can extrapolate — it can only predict within the range of training data

6 · When to Use What

  • 🔹 Random Forest → general-purpose, works well out of the box, handles nonlinearity and interactions
  • 🔹 Single Decision Tree → when interpretability is paramount (Project 12)
  • 🔹 Ridge/Lasso → when you need a linear model with coefficient interpretation (Projects 17-18)
  • 🔹 Logistic Regression → when you need probability calibration and simplicity (Project 10)
  • 🔹 OOB score → quick validation estimate, especially useful with small datasets

7 · When NOT to Use

  • 🚫 When you need a model that explains WHY — Random Forest is harder to interpret than linear models
  • 🚫 For extrapolation — Random Forest cannot predict values outside the training range
  • 🚫 When training time matters — 500 trees take 500× longer than one tree
  • 🚫 When you need very fast inference — each prediction requires traversing all trees

8 · How It Only Works — Internal Mechanics

Bagging — The Foundation

  • Create B bootstrap samples (random samples with replacement from training data)
  • Train one tree on each bootstrap sample
  • For classification: majority vote. For regression: average predictions
  • Key insight: averaging reduces variance without increasing bias

Random Feature Subsets — What Makes It a "Random" Forest

  • At each split, only consider max_features random features (default: √p for classification, p/3 for regression)
  • This decorrelates the trees — if one feature dominates, not every tree uses it at the top
  • More diverse trees → errors cancel out more effectively when averaged

Out-of-Bag (OOB) Error

  • Each bootstrap sample uses ~63.2% of the data (some samples are repeated)
  • The remaining ~36.8% are "out-of-bag" for that tree
  • For each sample, predict using only trees where it was OOB → free validation estimate
  • OOB error ≈ cross-validation error — no need for a separate validation set

Feature Importance — Mean Decrease in Impurity

  • For each feature, sum the total reduction in Gini impurity (Project 12) across all splits in all trees
  • Features that create the best splits get higher importance scores
  • Normalized to sum to 1.0
  • Caveat: biased toward high-cardinality features — permutation importance is more reliable (covered in Project 41)

Why More Trees Don't Overfit

  • Each additional tree is trained on a different bootstrap sample
  • Adding more trees reduces variance (averaging effect) without changing bias
  • Unlike polynomial degree (Project 20), n_estimators has no overfitting risk — only diminishing returns

9 · Common Pitfalls and Failure Modes

  • ⚠️ Too few trees — variance reduction requires many trees; 100 is a minimum
  • ⚠️ Trusting feature importance blindly — correlated features split importance between them
  • ⚠️ Not setting random_state — results vary between runs (fine for production, bad for debugging)
  • ⚠️ Using Random Forest for extrapolation — it predicts the nearest training value, not beyond
  • ⚠️ Ignoring max_depth — very deep trees in the forest can still memorize noise; consider limiting depth

10 · Scalability Considerations

  • Training: O(B × n × p × log(n)) where B = number of trees — parallelizable with n_jobs=-1
  • Prediction: O(B × depth) per sample — all trees must be traversed
  • Memory: stores all B trees — can be large with many deep trees
  • n_jobs=-1 uses all CPU cores — Random Forest is embarrassingly parallel
  • For very large datasets, consider subsampling (max_samples parameter) or fewer trees

11 · Interpretability Considerations

  • ✅ Feature importance gives a global ranking of feature relevance
  • ✅ OOB score provides a trustworthy performance estimate
  • ⚠️ Individual predictions are hard to explain — "500 trees voted, 340 said default"
  • ⚠️ Feature importance doesn't tell you the direction of the effect (positive or negative)
  • ⚠️ For per-prediction explanations, use SHAP values (covered in Project 27)

12 · Complete Working Code

import numpy as np
from sklearn.ensemble import RandomForestClassifier
from sklearn.tree import DecisionTreeClassifier
from sklearn.model_selection import cross_val_score, StratifiedKFold
from sklearn.metrics import classification_report, accuracy_score, f1_score
from sklearn.datasets import make_classification
import matplotlib
matplotlib.use('Agg')
import matplotlib.pyplot as plt

# ─── Step 1: Synthetic credit scoring data ───
X, y = make_classification(n_samples=2000, n_features=10, n_informative=6,
                           n_redundant=2, n_clusters_per_class=2,
                           flip_y=0.05, random_state=42)
feature_names = ['income', 'debt_ratio', 'credit_history', 'open_accounts',
                 'inquiries', 'employment_yrs', 'loan_amount', 'age',
                 'redundant_1', 'redundant_2']

print(f"=== Credit Scoring Dataset ===")
print(f"  Samples: {len(y)}, Features: {X.shape[1]}")
print(f"  Default rate: {np.mean(y):.1%}")

# ─── Step 2: Single tree instability ───
print(f"\n=== Single Tree Instability ===")
for seed in range(5):
    tree = DecisionTreeClassifier(random_state=seed)
    scores = cross_val_score(tree, X, y, cv=5, scoring='f1')
    print(f"  Seed {seed}: F1 = {scores.mean():.4f} ± {scores.std():.4f}")

# ─── Step 3: Random Forest — stable and better ───
print(f"\n=== Random Forest Stability ===")
for seed in range(5):
    rf = RandomForestClassifier(n_estimators=200, random_state=seed, n_jobs=-1)
    scores = cross_val_score(rf, X, y, cv=5, scoring='f1')
    print(f"  Seed {seed}: F1 = {scores.mean():.4f} ± {scores.std():.4f}")

# ─── Step 4: Single tree vs Forest comparison ───
print(f"\n=== Head-to-Head: Single Tree vs Random Forest ===")
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

models = [
    ('Single Tree (no limit)', DecisionTreeClassifier(random_state=42)),
    ('Single Tree (depth=5)', DecisionTreeClassifier(max_depth=5, random_state=42)),
    ('RF (10 trees)', RandomForestClassifier(n_estimators=10, random_state=42, n_jobs=-1)),
    ('RF (100 trees)', RandomForestClassifier(n_estimators=100, random_state=42, n_jobs=-1)),
    ('RF (500 trees)', RandomForestClassifier(n_estimators=500, random_state=42, n_jobs=-1)),
]

for name, model in models:
    scores = cross_val_score(model, X, y, cv=cv, scoring='f1')
    print(f"  {name:>25}: F1 = {scores.mean():.4f} ± {scores.std():.4f}")

# ─── Step 5: OOB score — free validation ───
rf_oob = RandomForestClassifier(n_estimators=300, oob_score=True, random_state=42, n_jobs=-1)
rf_oob.fit(X, y)
print(f"\n=== Out-of-Bag Score ===")
print(f"  OOB accuracy: {rf_oob.oob_score_:.4f}")
print(f"  (This is a free validation estimate — no separate test set needed)")

# ─── Step 6: Feature importance ───
importances = rf_oob.feature_importances_
sorted_idx = np.argsort(importances)[::-1]

print(f"\n=== Feature Importance (Mean Decrease in Impurity) ===")
for i in sorted_idx:
    bar = "█" * int(importances[i] * 50)
    print(f"  {feature_names[i]:>16}: {importances[i]:.4f} {bar}")

# ─── Step 7: Visualization ───
fig, axes = plt.subplots(1, 3, figsize=(18, 5))

# Plot 1: Feature importance
axes[0].barh(range(len(feature_names)), importances[sorted_idx], color='#58a6ff', alpha=0.8)
axes[0].set_yticks(range(len(feature_names)))
axes[0].set_yticklabels([feature_names[i] for i in sorted_idx])
axes[0].set_xlabel('Importance')
axes[0].set_title('Feature Importance')
axes[0].grid(True, alpha=0.2, axis='x')

# Plot 2: n_estimators vs OOB error
n_trees_range = [1, 5, 10, 25, 50, 100, 200, 300, 500]
oob_errors = []
for n in n_trees_range:
    rf_temp = RandomForestClassifier(n_estimators=n, oob_score=True,
                                      random_state=42, n_jobs=-1)
    rf_temp.fit(X, y)
    oob_errors.append(1 - rf_temp.oob_score_)

axes[1].plot(n_trees_range, oob_errors, 'o-', color='#f0883e', linewidth=2)
axes[1].set_xlabel('Number of Trees')
axes[1].set_ylabel('OOB Error Rate')
axes[1].set_title('More Trees → Lower Error (Diminishing Returns)')
axes[1].grid(True, alpha=0.2)

# Plot 3: Single tree vs Forest variance
n_runs = 20
tree_scores = []
rf_scores = []
for seed in range(n_runs):
    tree = DecisionTreeClassifier(random_state=seed)
    scores_t = cross_val_score(tree, X, y, cv=3, scoring='f1')
    tree_scores.append(scores_t.mean())

    rf = RandomForestClassifier(n_estimators=100, random_state=seed, n_jobs=-1)
    scores_r = cross_val_score(rf, X, y, cv=3, scoring='f1')
    rf_scores.append(scores_r.mean())

axes[2].boxplot([tree_scores, rf_scores], labels=['Single Tree', 'Random Forest (100)'])
axes[2].set_ylabel('F1 Score')
axes[2].set_title('Variance: Single Tree vs Forest')
axes[2].grid(True, alpha=0.2, axis='y')

plt.tight_layout()
plt.savefig('project_21_random_forest.png', dpi=100, bbox_inches='tight')
print(f"\n✅ Plots saved as project_21_random_forest.png")

13 · Key Takeaway + Bridge to Next Project

Key Takeaway

  • Random Forest = many decision trees trained on random data subsets with random feature subsets
  • Averaging many diverse trees reduces variance without increasing bias
  • OOB score gives a free validation estimate — each sample is predicted by trees that didn't train on it
  • Feature importance ranks which features drive predictions (but doesn't show direction)
  • More trees rarely hurt — only diminishing returns and slower training
  • No feature scaling needed — tree-based models are scale-invariant
