Federated Learning FedAvg Data Privacy Simulation

Difficulty: Very Hard  |  Builds on: Projects 47, 55

1 · Hypothetical Problem

  • Three hospitals each have patient records for predicting diabetes onset — Hospital A has 2,000 patients, Hospital B has 1,500, Hospital C has 800
  • A centralized model trained on all 4,300 patients would be the most accurate, but HIPAA and GDPR regulations prohibit sharing raw patient data
  • Each hospital training its own local model suffers from small sample sizes and biased demographics
  • Hospital A serves mostly elderly patients, Hospital B has younger urban patients, Hospital C is a rural clinic with diverse ages
  • Goal → use federated learning to train a shared model WITHOUT any hospital sharing raw patient data — only model updates are exchanged

2 · How to Approach It

  1. From Project 47: we built neural networks from scratch — federated learning extends this to distributed training
  2. From Project 55: we built ML pipelines — federated learning is a distributed pipeline pattern
  3. The key insight: instead of sending data to the model, send the model to the data
  4. Each hospital trains a local copy of the model on its own data, then sends only the model weights (not data) to a central server
  5. The server averages the weights and sends the updated global model back — this is the FedAvg algorithm
  6. We'll simulate this entire process using numpy to understand the mechanics without any deep learning framework

3 · How Can We Solve It

  1. Generate synthetic patient data for 3 hospitals with different demographic distributions
  2. Implement a simple logistic regression model using numpy (weights, bias, sigmoid, gradient descent)
  3. Implement the FedAvg algorithm: local training → weight upload → server averaging → weight download
  4. Run federated training for multiple communication rounds
  5. Compare three approaches: (a) centralized (all data pooled), (b) federated (FedAvg), (c) local-only (each hospital alone)
  6. Analyze convergence speed, final accuracy, and fairness across hospitals

4 · What Should Do

  • ✅ Weight the averaging by each client's dataset size — larger hospitals should have more influence
  • ✅ Run multiple local epochs before each communication round — reduces communication overhead
  • ✅ Evaluate the global model on each hospital's data separately — check for fairness across clients
  • ✅ Monitor convergence — federated learning typically needs more rounds than centralized training
  • ✅ Consider non-IID data — real-world clients have different data distributions; test robustness
  • ✅ Use secure aggregation in production — even model updates can leak information about training data

5 · What Should NOT Do

  • ❌ Do NOT share raw data between clients — that defeats the entire purpose of federated learning
  • ❌ Do NOT assume IID data across clients — hospitals serve different populations; this is the hard part
  • ❌ Do NOT ignore communication costs — each round requires sending full model weights; minimize rounds
  • ❌ Do NOT trust model updates blindly — a malicious client can poison the global model (Byzantine attacks)
  • ❌ Do NOT use simple averaging when client datasets are very different sizes — weighted averaging is essential
  • ❌ Do NOT skip local evaluation — the global model might perform well on average but poorly for one hospital

6 · When to Use What

  • 🔹 Federated Learning (FedAvg) → multiple data owners, privacy constraints, similar tasks across clients
  • 🔹 Centralized Training → when data can be pooled; always the performance upper bound
  • 🔹 Local-Only Training → when data is sufficient per client and tasks are truly different
  • 🔹 Differential Privacy → add noise to model updates for mathematical privacy guarantees
  • 🔹 Split Learning → when even model updates are too sensitive; split the model between client and server
  • 🔹 Transfer Learning (Project 48) → when one client has lots of data and others have little

7 · When NOT to Use

  • 🚫 When data CAN be centralized — federated learning adds complexity for no benefit
  • 🚫 When clients have extremely different tasks — FedAvg assumes a shared model architecture and objective
  • 🚫 When one client has 99% of the data — just train on that client; federation adds overhead
  • 🚫 When communication bandwidth is extremely limited — each round sends full model weights
  • 🚫 When you need mathematical privacy guarantees — FedAvg alone doesn't provide differential privacy

8 · How It Only Works — Internal Mechanics

The FedAvg Algorithm

  • Initialize a global model with random weights W₀
  • For each communication round t = 1, 2, ..., T:
  •   1. Server sends global weights Wₜ to all K clients
  •   2. Each client k trains locally for E epochs on its own data → produces local weights Wₜᵏ
  •   3. Each client sends Wₜᵏ back to the server
  •   4. Server computes weighted average: Wₜ₊₁ = Σ (nₖ/n) × Wₜᵏ where nₖ is client k's dataset size
  • After T rounds, the global model Wₜ has learned from all clients' data without seeing any raw data

Why Weighted Averaging Works

  • Each client's gradient update points toward the local optimum for that client's data
  • Averaging gradients (or weights after gradient steps) approximates the gradient on the combined dataset
  • With IID data, FedAvg converges to the same solution as centralized SGD
  • With non-IID data, local models diverge — more communication rounds are needed to keep them aligned

Non-IID Challenge

  • IID: each client's data is a random sample from the same distribution — easy case
  • Non-IID: clients have different distributions (e.g., different age groups, different disease prevalence)
  • Non-IID causes "client drift" — local models optimize for different objectives
  • Solutions: more frequent communication, smaller local learning rates, FedProx (adds proximal term)

Privacy Considerations

  • Raw data never leaves the client — only model weights/gradients are shared
  • But model updates CAN leak information — gradient inversion attacks can reconstruct training data
  • Secure aggregation: server only sees the sum of updates, not individual client updates
  • Differential privacy: add calibrated noise to updates for mathematical privacy guarantees

9 · Common Pitfalls and Failure Modes

  • ⚠️ Non-IID data divergence — local models drift apart; global model oscillates instead of converging
  • ⚠️ Straggler problem — one slow client delays the entire round; use asynchronous aggregation
  • ⚠️ Communication bottleneck — sending full model weights each round is expensive for large models
  • ⚠️ Free-rider problem — a client contributes no useful updates but benefits from the global model
  • ⚠️ Poisoning attacks — a malicious client sends corrupted weights to degrade the global model
  • ⚠️ Overfitting to local data — too many local epochs before communication causes divergence

10 · Scalability Considerations

  • Communication cost: O(T × K × |W|) where T = rounds, K = clients, |W| = model size
  • Gradient compression: send only top-k% of gradient values to reduce bandwidth by 100×
  • Client selection: randomly sample a subset of clients each round (partial participation)
  • Asynchronous FL: don't wait for all clients — aggregate as updates arrive
  • Model distillation: clients send predictions instead of weights — smaller communication payload
  • Real-world scale: Google's keyboard prediction uses FL with millions of mobile devices

11 · Interpretability Considerations

  • ✅ The global model is interpretable in the same way as any centralized model — same architecture
  • ✅ Per-client evaluation reveals fairness — does the model work equally well for all hospitals?
  • ⚠️ Feature importance may differ across clients — the global model is a compromise
  • ⚠️ Debugging is harder — you can't inspect the training data when something goes wrong
  • ✅ Convergence curves show whether clients are aligned or diverging — useful diagnostic
  • ⚠️ Explaining WHY the model makes a prediction requires local data access — SHAP values need data

def generate_hospital_data(n_samples, age_mean, bmi_mean, glucose_mean, label_noise=0.1):
    """Generate synthetic patient data for one hospital."""
    age = np.random.normal(age_mean, 8, n_samples)
    bmi = np.random.normal(bmi_mean, 4, n_samples)
    glucose = np.random.normal(glucose_mean, 20, n_samples)
    blood_pressure = np.random.normal(80, 12, n_samples)
    insulin = np.random.exponential(100, n_samples)
    skin_thickness = np.random.normal(25, 8, n_samples)

    X = np.column_stack([age, bmi, glucose, blood_pressure, insulin, skin_thickness])

    # Diabetes probability based on features
    log_odds = (-5 + 0.03 * age + 0.08 * bmi + 0.02 * glucose
                + 0.01 * blood_pressure + 0.002 * insulin)
    prob = 1 / (1 + np.exp(-log_odds))
    y = (np.random.random(n_samples) < prob).astype(float)

    return X, y

# STEP 2: Logistic Regression Model (NumPy from scratch)
# ═══════════════════════════════════════════════════════════════
class LogisticRegressionNumpy:
    """Simple logistic regression implemented in pure numpy."""

    def __init__(self, n_features):
        self.weights = np.zeros(n_features)
        self.bias = 0.0

    def sigmoid(self, z):
        return 1 / (1 + np.exp(-np.clip(z, -500, 500)))

    def predict_proba(self, X):
        return self.sigmoid(X @ self.weights + self.bias)

    def predict(self, X):
        return (self.predict_proba(X) >= 0.5).astype(float)

    def compute_loss(self, X, y):
        probs = self.predict_proba(X)
        eps = 1e-7
        return -np.mean(y * np.log(probs + eps) + (1 - y) * np.log(1 - probs + eps))

    def train_step(self, X, y, lr=0.1):
        """One epoch of gradient descent."""
        n = len(y)
        probs = self.predict_proba(X)
        error = probs - y

        grad_w = (X.T @ error) / n
        grad_b = np.mean(error)

        self.weights -= lr * grad_w
        self.bias -= lr * grad_b

        return self.compute_loss(X, y)

    def get_weights(self):
        return self.weights.copy(), self.bias

    def set_weights(self, weights, bias):
        self.weights = weights.copy()
        self.bias = bias

# ═══════════════════════════════════════════════════════════════
# STEP 3: FedAvg Algorithm
# ═══════════════════════════════════════════════════════════════
def federated_averaging(client_data, n_rounds=30, local_epochs=5, lr=0.1):
    """
    Federated Averaging (FedAvg) algorithm.
    client_data: list of (X, y) tuples for each client
    """
    n_features = client_data[0][0].shape[1]
    n_clients = len(client_data)
    total_samples = sum(len(y) for _, y in client_data)

    # Initialize global model
    global_model = LogisticRegressionNumpy(n_features)

    # Track metrics
    history = {'round': [], 'global_loss': [], 'global_acc': [],
               'client_losses': [], 'client_accs': []}

    print(f"\n=== Federated Learning — {n_clients} Clients, {n_rounds} Rounds ===")

    for round_num in range(n_rounds):
        global_weights, global_bias = global_model.get_weights()
        client_weights = []
        client_biases = []
        client_n_samples = []
        round_losses = []
        round_accs = []

        # Each client trains locally
        for k, (X_k, y_k) in enumerate(client_data):
            # Download global model
            local_model = LogisticRegressionNumpy(n_features)
            local_model.set_weights(global_weights, global_bias)

            # Local training
            for epoch in range(local_epochs):
                loss = local_model.train_step(X_k, y_k, lr=lr)

            # Upload local weights
            w_k, b_k = local_model.get_weights()
            client_weights.append(w_k)
            client_biases.append(b_k)
            client_n_samples.append(len(y_k))

            # Track local metrics
            round_losses.append(loss)
            round_accs.append(accuracy_score(y_k, local_model.predict(X_k)))

        # Server: weighted averaging
        new_weights = np.zeros(n_features)
        new_bias = 0.0
        for k in range(n_clients):
            weight_factor = client_n_samples[k] / total_samples
            new_weights += weight_factor * client_weights[k]
            new_bias += weight_factor * client_biases[k]

        global_model.set_weights(new_weights, new_bias)

        # Evaluate global model on all data
        X_combined = np.vstack([X for X, _ in client_data])
        y_combined = np.concatenate([y for _, y in client_data])
        global_loss = global_model.compute_loss(X_combined, y_combined)
        global_acc = accuracy_score(y_combined, global_model.predict(X_combined))

        history['round'].append(round_num + 1)
        history['global_loss'].append(global_loss)
        history['global_acc'].append(global_acc)
        history['client_losses'].append(round_losses)
        history['client_accs'].append(round_accs)

        if (round_num + 1) % 10 == 0 or round_num == 0:
            print(f"  Round {round_num+1:>3}: Global Loss={global_loss:.4f}, "
                  f"Global Acc={global_acc:.4f}, "
                  f"Client Accs=[{', '.join(f'{a:.3f}' for a in round_accs)}]")

    return global_model, history

# Run federated learning
