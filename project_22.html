Project 22: Tuning Hyperparameters — GridSearchCV vs RandomizedSearchCV

Hyperparameter Tuning GridSearchCV RandomizedSearchCV param_grid Nested CV

Difficulty: Medium  |  Builds on: Projects 15, 21

1 · Hypothetical Problem

  • You've built a Random Forest for credit scoring (Project 21) — it works, but is it the best it can be?
  • The model has many knobs: n_estimators, max_depth, max_features, min_samples_split, min_samples_leaf
  • You tried a few values manually, but there are thousands of possible combinations
  • Goal → systematically search the hyperparameter space to find the optimal configuration

2 · How to Approach It

  1. Define the hyperparameter space — which parameters to tune and what values to try
  2. GridSearchCV: try every combination (exhaustive but expensive)
  3. RandomizedSearchCV: sample random combinations (faster, often finds good solutions)
  4. Both use cross-validation (Project 15) internally to evaluate each combination
  5. For honest performance estimation, use nested CV — inner loop tunes, outer loop evaluates

3 · How Can We Solve It

  1. Set up a Random Forest with a hyperparameter grid
  2. Run GridSearchCV — see all combinations evaluated
  3. Run RandomizedSearchCV — see how it finds good solutions faster
  4. Compare the best parameters and scores from both approaches
  5. Demonstrate nested CV for unbiased performance estimation
  6. Visualize the search results — which parameters matter most?

4 · What Should Do

  • ✅ Start with RandomizedSearchCV for a broad search, then refine with GridSearchCV
  • ✅ Use StratifiedKFold for classification tasks (Project 15)
  • ✅ Choose the right scoring metric — F1 for imbalanced data (Project 19), RMSE for regression
  • ✅ Use nested CV when reporting final performance — avoids optimistic bias
  • ✅ Check cv_results_ to understand which parameters matter most

5 · What Should NOT Do

  • ❌ Do NOT report the best score from GridSearchCV as your model's performance — it's optimistically biased
  • ❌ Do NOT use GridSearchCV with a huge parameter space — 5 params × 10 values each = 100,000 combinations × 5 folds = 500,000 fits
  • ❌ Do NOT tune hyperparameters on the test set — that's data leakage
  • ❌ Do NOT forget to set random_state for reproducibility
  • ❌ Do NOT tune parameters that don't matter — n_estimators beyond 200 rarely changes results

6 · When to Use What

  • 🔹 GridSearchCV → small parameter space (≤100 combinations), when you want exhaustive search
  • 🔹 RandomizedSearchCV → large parameter space, continuous parameters, when time is limited
  • 🔹 Nested CV → when you need an unbiased estimate of the tuned model's performance
  • 🔹 Manual tuning → for quick experiments or when you know the model well
  • 🔹 Bayesian optimization (Optuna) → for expensive models where each evaluation is costly (covered in Project 42)

7 · When NOT to Use

  • 🚫 GridSearchCV with more than ~1000 combinations — too slow, use RandomizedSearchCV
  • 🚫 When the model is already performing well — diminishing returns from tuning
  • 🚫 When the dataset is tiny — cross-validation folds become too small to be reliable
  • 🚫 When you should be improving features instead — better features beat better hyperparameters

8 · How It Only Works — Internal Mechanics

GridSearchCV — Exhaustive Search

  • Takes a param_grid dictionary: each key is a parameter, each value is a list of options
  • Generates the Cartesian product of all parameter combinations
  • For each combination, runs K-fold cross-validation and records the mean score
  • Returns the combination with the best mean CV score
  • Total fits = (number of combinations) × K folds

RandomizedSearchCV — Random Sampling

  • Takes a param_distributions dictionary — can include continuous distributions (not just lists)
  • Samples n_iter random combinations from the parameter space
  • For each sampled combination, runs K-fold CV
  • Key advantage: with 1000 possible combinations, 60 random samples have a 95% chance of finding a top-5% solution
  • Supports scipy distributions: randint(10, 500), uniform(0.01, 1.0)

Why RandomizedSearchCV Often Wins

  • GridSearch wastes time on unimportant parameters — if max_depth doesn't matter, it still tries all values
  • RandomSearch explores more values of important parameters by chance
  • With the same compute budget, RandomSearch covers more of the space that matters

Nested Cross-Validation — Unbiased Estimation

  • Problem: the best score from GridSearchCV is optimistic — it selected the best from many options
  • Solution: outer CV loop evaluates the entire tuning process as a single "model"
  • Inner loop: GridSearchCV finds best params on training fold
  • Outer loop: evaluates the tuned model on the held-out fold
  • The outer score is an unbiased estimate of the tuned model's true performance

9 · Common Pitfalls and Failure Modes

  • ⚠️ Reporting GridSearchCV best_score_ as final performance — it's optimistically biased; use nested CV
  • ⚠️ Combinatorial explosion — 5 params × 10 values = 100K combinations; use RandomizedSearchCV
  • ⚠️ Tuning on the test set — if you peek at test performance to guide tuning, your estimate is invalid
  • ⚠️ Overfitting the validation set — with enough tuning iterations, you can overfit to the CV folds
  • ⚠️ Ignoring the search results — cv_results_ tells you which parameters actually matter

10 · Scalability Considerations

  • GridSearchCV: O(combinations × K × model_training_time) — grows multiplicatively
  • RandomizedSearchCV: O(n_iter × K × model_training_time) — you control the budget
  • Use n_jobs=-1 to parallelize across CPU cores
  • For expensive models, start with RandomizedSearchCV (n_iter=50), then GridSearchCV around the best region
  • Consider HalvingGridSearchCV (sklearn) — starts with small data, eliminates bad candidates early

11 · Interpretability Considerations

  • ✅ cv_results_ shows how each parameter affects performance — useful for understanding the model
  • ✅ You can plot parameter vs score to see which parameters matter
  • ⚠️ The "best" parameters are specific to this dataset — they may not transfer to new data
  • ⚠️ Small differences in CV score (e.g., 0.001) are likely noise — don't over-optimize

12 · Complete Working Code

import numpy as np
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import (GridSearchCV, RandomizedSearchCV,
                                      StratifiedKFold, cross_val_score)
from sklearn.datasets import make_classification
from sklearn.metrics import f1_score, make_scorer
from scipy.stats import randint, uniform
import matplotlib
matplotlib.use('Agg')
import matplotlib.pyplot as plt
import time

# ─── Step 1: Credit scoring data (same problem as Project 21) ───
X, y = make_classification(n_samples=2000, n_features=10, n_informative=6,
                           n_redundant=2, n_clusters_per_class=2,
                           flip_y=0.05, random_state=42)

print(f"=== Dataset: {X.shape[0]} samples, {X.shape[1]} features ===")

# ─── Step 2: GridSearchCV — exhaustive search ───
param_grid = {
    'n_estimators': [50, 100, 200],
    'max_depth': [5, 10, 20, None],
    'max_features': ['sqrt', 'log2', 0.5],
    'min_samples_split': [2, 5, 10],
}

total_combinations = 1
for v in param_grid.values():
    total_combinations *= len(v)
print(f"\n=== GridSearchCV ===")
print(f"  Parameter combinations: {total_combinations}")
print(f"  Total fits (5-fold CV): {total_combinations * 5}")

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

start = time.time()
grid_search = GridSearchCV(
    RandomForestClassifier(random_state=42, n_jobs=-1),
    param_grid=param_grid,
    cv=cv,
    scoring='f1',
    n_jobs=-1,
    return_train_score=True
)
grid_search.fit(X, y)
grid_time = time.time() - start

print(f"  Time: {grid_time:.1f}s")
print(f"  Best score (CV F1): {grid_search.best_score_:.4f}")
print(f"  Best params: {grid_search.best_params_}")

# ─── Step 3: RandomizedSearchCV — random sampling ───
param_distributions = {
    'n_estimators': randint(50, 500),
    'max_depth': [5, 10, 15, 20, 30, None],
    'max_features': uniform(0.1, 0.9),
    'min_samples_split': randint(2, 20),
    'min_samples_leaf': randint(1, 10),
}

print(f"\n=== RandomizedSearchCV ===")
print(f"  Sampling 60 random combinations from continuous distributions")

start = time.time()
random_search = RandomizedSearchCV(
    RandomForestClassifier(random_state=42, n_jobs=-1),
    param_distributions=param_distributions,
    n_iter=60,
    cv=cv,
    scoring='f1',
    n_jobs=-1,
    random_state=42,
    return_train_score=True
)
random_search.fit(X, y)
rand_time = time.time() - start

print(f"  Time: {rand_time:.1f}s")
print(f"  Best score (CV F1): {random_search.best_score_:.4f}")
print(f"  Best params: {random_search.best_params_}")

# ─── Step 4: Compare search results ───
print(f"\n=== Comparison ===")
print(f"  {'Method':<20} {'Best F1':>10} {'Time':>10} {'Fits':>10}")
print(f"  {'-'*50}")
print(f"  {'GridSearchCV':<20} {grid_search.best_score_:>10.4f} {grid_time:>9.1f}s {total_combinations*5:>10}")
print(f"  {'RandomizedSearchCV':<20} {random_search.best_score_:>10.4f} {rand_time:>9.1f}s {60*5:>10}")

# ─── Step 5: Nested CV — unbiased performance estimate ───
print(f"\n=== Nested CV: Unbiased Performance Estimate ===")
outer_cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=0)
inner_cv = StratifiedKFold(n_splits=3, shuffle=True, random_state=1)

# Inner loop: tune hyperparameters
inner_search = GridSearchCV(
    RandomForestClassifier(random_state=42, n_jobs=-1),
    param_grid={'n_estimators': [100, 200], 'max_depth': [10, 20, None],
                'max_features': ['sqrt', 0.5]},
    cv=inner_cv, scoring='f1', n_jobs=-1
)

# Outer loop: evaluate the tuned model
nested_scores = cross_val_score(inner_search, X, y, cv=outer_cv, scoring='f1')
print(f"  Nested CV F1: {nested_scores.mean():.4f} ± {nested_scores.std():.4f}")
print(f"  GridSearchCV best_score_: {grid_search.best_score_:.4f} ← slightly optimistic")

# ─── Step 6: Analyze which parameters matter ───
print(f"\n=== Parameter Impact Analysis (from GridSearchCV) ===")
results = grid_search.cv_results_

fig, axes = plt.subplots(2, 2, figsize=(14, 10))

# Plot each parameter's effect on score
for ax, param_name in zip(axes.ravel(),
                           ['param_max_depth', 'param_n_estimators',
                            'param_max_features', 'param_min_samples_split']):
    param_values = results[param_name].data
    mean_scores = results['mean_test_score']

    unique_vals = sorted(set(str(v) for v in param_values))
    val_scores = {v: [] for v in unique_vals}
    for v, s in zip(param_values, mean_scores):
        val_scores[str(v)].append(s)

    positions = range(len(unique_vals))
    bp = ax.boxplot([val_scores[v] for v in unique_vals], positions=list(positions),
                     patch_artist=True)
    for patch in bp['boxes']:
        patch.set_facecolor('#58a6ff')
        patch.set_alpha(0.5)
    ax.set_xticks(list(positions))
    ax.set_xticklabels(unique_vals, rotation=45)
    ax.set_title(param_name.replace('param_', ''))
    ax.set_ylabel('CV F1 Score')
    ax.grid(True, alpha=0.2, axis='y')

plt.suptitle('How Each Hyperparameter Affects Performance', fontsize=13, color='#c9d1d9')
plt.tight_layout()
plt.savefig('project_22_param_analysis.png', dpi=100, bbox_inches='tight')
print(f"\n✅ Parameter analysis saved as project_22_param_analysis.png")

13 · Key Takeaway + Bridge to Next Project

Key Takeaway

  • GridSearchCV tries every combination — exhaustive but expensive
  • RandomizedSearchCV samples random combinations — faster, often finds equally good solutions
  • The best_score_ from either search is optimistically biased — use nested CV for honest estimates
  • Start broad with RandomizedSearchCV, then refine with GridSearchCV around the best region
  • Analyze cv_results_ to understand which parameters actually matter
