Project 20: Predicting Next Month's Revenue — Polynomial Features

Polynomial Regression PolynomialFeatures Degree Selection Nonlinear Patterns Overfitting

Difficulty: Medium  |  Builds on: Projects 8, 17

1 · Hypothetical Problem

  • You're a financial analyst predicting next month's revenue for a growing SaaS company
  • Revenue has been growing — but not linearly. Early months show slow growth, then acceleration, then a plateau
  • A straight line (linear regression from Project 2) misses the curve entirely
  • Features: months since launch, marketing spend, number of active users
  • Goal → capture the nonlinear growth pattern using polynomial features

2 · How to Approach It

  1. Recognize the pattern: residuals from linear regression show a systematic curve (Project 5)
  2. Polynomial features transform x into [x, x², x³, ...] — still a linear model, but on nonlinear features
  3. Choose the degree carefully — too low underfits, too high overfits (Project 8's bias-variance tradeoff)
  4. Combine with regularization (Project 17) to control overfitting at higher degrees
  5. Use cross-validation (Project 15) to select the best degree

3 · How Can We Solve It

  1. Generate revenue data with a true nonlinear pattern
  2. Fit linear regression — show it misses the curve
  3. Apply PolynomialFeatures with degrees 2, 3, 5, 10, 15
  4. Visualize each fit — watch overfitting emerge at high degrees
  5. Use cross-validation to find the optimal degree
  6. Add Ridge regularization to tame high-degree polynomials

4 · What Should Do

  • ✅ Start with low degree (2 or 3) and increase only if needed
  • ✅ Use cross-validation to select degree — not just training error
  • ✅ Scale features before polynomial expansion — large values raised to high powers explode
  • ✅ Combine polynomial features with Ridge/Lasso to regularize (Projects 17, 18)
  • ✅ Plot the fit visually — polynomial overfitting is easy to spot by eye

5 · What Should NOT Do

  • ❌ Do NOT use high-degree polynomials (>5) without regularization — they overfit wildly
  • ❌ Do NOT forget to scale before polynomial expansion — x=1000 becomes x¹⁰=10³⁰
  • ❌ Do NOT extrapolate with polynomial models — they diverge rapidly outside the training range
  • ❌ Do NOT confuse polynomial regression with nonlinear models — it's still linear in the parameters
  • ❌ Do NOT use polynomial features with many input features — the number of terms explodes combinatorially

6 · When to Use What

  • 🔹 Degree 2 → captures simple curves (U-shapes, diminishing returns)
  • 🔹 Degree 3 → captures S-curves and inflection points
  • 🔹 Degree 4+ → rarely needed; consider other models instead
  • 🔹 Polynomial + Ridge → when you want flexibility but fear overfitting
  • 🔹 interaction_only=True → when you want feature interactions (x₁×x₂) without powers

7 · When NOT to Use

  • 🚫 With many features — 10 features at degree 3 creates 286 polynomial terms
  • 🚫 For extrapolation — polynomial predictions outside training range are unreliable
  • 🚫 When the nonlinearity is better captured by tree-based models (Project 21)
  • 🚫 When the relationship is truly linear — adding polynomial terms just adds noise

8 · How It Only Works — Internal Mechanics

What PolynomialFeatures Actually Does

  • Input: [x₁, x₂] with degree=2
  • Output: [1, x₁, x₂, x₁², x₁x₂, x₂²] — all combinations up to degree 2
  • The model is still linear: ŷ = w₀ + w₁x₁ + w₂x₂ + w₃x₁² + w₄x₁x₂ + w₅x₂²
  • Linear in parameters (w), nonlinear in features (x) — that's the trick

Feature Explosion

  • With p features and degree d, the number of polynomial terms is: C(p+d, d)
  • p=5, d=2 → 21 terms. p=5, d=5 → 252 terms. p=10, d=3 → 286 terms
  • This is why polynomial features + many input features = disaster without regularization

Why High Degree Overfits

  • A degree-n polynomial can pass through n+1 points exactly
  • With enough degree, the model memorizes training data — including noise
  • Between training points, the polynomial oscillates wildly (Runge's phenomenon)
  • This is the bias-variance tradeoff from Project 8 in action: low degree = high bias, high degree = high variance

Regularization Saves High-Degree Polynomials

  • Ridge penalty (Project 17) shrinks polynomial coefficients — prevents wild oscillations
  • You can use a high degree but let regularization control which terms matter
  • This is more flexible than manually choosing the degree

9 · Common Pitfalls and Failure Modes

  • ⚠️ Numerical overflow — x=100 at degree 10 = 10²⁰. Always scale first
  • ⚠️ Overfitting with high degree — training R² = 0.99, test R² = -5.0 (worse than predicting the mean)
  • ⚠️ Extrapolation disaster — polynomial predictions outside training range diverge to ±∞
  • ⚠️ Feature explosion with multiple inputs — 10 features at degree 5 = 3003 terms
  • ⚠️ Choosing degree by training error — always use cross-validation (Project 15)

10 · Scalability Considerations

  • Feature count grows combinatorially with degree — memory and compute scale accordingly
  • For 1-2 features, polynomial regression is fast even at high degrees
  • For many features, keep degree ≤ 2 or use interaction_only=True
  • Consider sparse polynomial features (most coefficients will be near zero with regularization)
  • Tree-based models (Project 21) handle nonlinearity without explicit feature engineering

11 · Interpretability Considerations

  • ✅ Degree 2 is interpretable: "revenue increases with spend, but with diminishing returns (negative x² term)"
  • ✅ Interaction terms are meaningful: "the effect of marketing depends on user count (x₁×x₂ term)"
  • ⚠️ Degree 3+ becomes hard to explain — "the cubic term of marketing spend" means little to stakeholders
  • ⚠️ With many polynomial terms, individual coefficients lose meaning — consider it a black box

12 · Complete Working Code

import numpy as np
from sklearn.linear_model import LinearRegression, Ridge
from sklearn.preprocessing import PolynomialFeatures, StandardScaler
from sklearn.pipeline import Pipeline
from sklearn.model_selection import cross_val_score, KFold
from sklearn.metrics import mean_squared_error, r2_score
import matplotlib
matplotlib.use('Agg')
import matplotlib.pyplot as plt

# ─── Step 1: Synthetic revenue data with nonlinear growth ───
np.random.seed(42)
n = 80
months = np.linspace(1, 36, n).reshape(-1, 1)

# True pattern: S-curve growth (logistic-like)
true_revenue = 500 + 200 * np.log(months) + 0.5 * months**2 - 0.005 * months**3
true_revenue = true_revenue.ravel()
noise = np.random.normal(0, 30, n)
revenue = true_revenue + noise

print(f"=== Revenue Data ===")
print(f"  Samples: {n}, Feature: months (1-36)")
print(f"  Revenue range: {revenue.min():.0f} to {revenue.max():.0f}")

# ─── Step 2: Linear regression — misses the curve ───
lr = LinearRegression().fit(months, revenue)
print(f"\n=== Linear Regression ===")
print(f"  Train R²: {lr.score(months, revenue):.4f}")

# ─── Step 3: Polynomial fits at different degrees ───
degrees = [1, 2, 3, 5, 10, 15]
fig, axes = plt.subplots(2, 3, figsize=(15, 10))
axes = axes.ravel()

months_plot = np.linspace(0, 40, 200).reshape(-1, 1)

for idx, deg in enumerate(degrees):
    pipe = Pipeline([
        ('scaler', StandardScaler()),
        ('poly', PolynomialFeatures(degree=deg)),
        ('lr', LinearRegression())
    ])
    pipe.fit(months, revenue)
    train_r2 = pipe.score(months, revenue)

    # CV score
    cv = KFold(n_splits=5, shuffle=True, random_state=42)
    cv_scores = cross_val_score(pipe, months, revenue, cv=cv, scoring='r2')

    y_plot = pipe.predict(months_plot)

    axes[idx].scatter(months, revenue, alpha=0.5, s=20, color='#58a6ff')
    axes[idx].plot(months_plot, y_plot, color='#f0883e', linewidth=2)
    axes[idx].set_title(f'Degree {deg}\nTrain R²={train_r2:.3f}, CV R²={cv_scores.mean():.3f}')
    axes[idx].set_xlabel('Months')
    axes[idx].set_ylabel('Revenue')
    axes[idx].set_ylim(revenue.min() - 100, revenue.max() + 100)
    axes[idx].grid(True, alpha=0.2)

plt.suptitle('Polynomial Regression: Underfitting → Overfitting', fontsize=14, color='#c9d1d9')
plt.tight_layout()
plt.savefig('project_20_polynomial_degrees.png', dpi=100, bbox_inches='tight')
print(f"\n✅ Degree comparison saved as project_20_polynomial_degrees.png")

# ─── Step 4: CV to find optimal degree ───
print(f"\n=== Cross-Validation: Finding Optimal Degree ===")
cv = KFold(n_splits=5, shuffle=True, random_state=42)
test_degrees = range(1, 16)
cv_means = []
cv_stds = []

for deg in test_degrees:
    pipe = Pipeline([
        ('scaler', StandardScaler()),
        ('poly', PolynomialFeatures(degree=deg)),
        ('lr', LinearRegression())
    ])
    scores = cross_val_score(pipe, months, revenue, cv=cv, scoring='neg_mean_squared_error')
    rmse = np.sqrt(-scores.mean())
    cv_means.append(rmse)
    cv_stds.append(np.sqrt(scores.std()))
    print(f"  Degree {deg:>2}: CV RMSE = {rmse:.2f}")

best_deg = test_degrees[np.argmin(cv_means)]
print(f"\n  → Best degree: {best_deg} (CV RMSE = {min(cv_means):.2f})")

# ─── Step 5: Polynomial + Ridge regularization ───
print(f"\n=== Polynomial (degree=10) + Ridge vs No Regularization ===")
pipe_no_reg = Pipeline([
    ('scaler', StandardScaler()),
    ('poly', PolynomialFeatures(degree=10)),
    ('lr', LinearRegression())
])
pipe_ridge = Pipeline([
    ('scaler', StandardScaler()),
    ('poly', PolynomialFeatures(degree=10)),
    ('ridge', Ridge(alpha=10))
])

for name, pipe in [('No regularization', pipe_no_reg), ('Ridge (α=10)', pipe_ridge)]:
    scores = cross_val_score(pipe, months, revenue, cv=cv, scoring='neg_mean_squared_error')
    rmse = np.sqrt(-scores.mean())
    pipe.fit(months, revenue)
    train_r2 = pipe.score(months, revenue)
    print(f"  {name:>20}: Train R² = {train_r2:.4f}, CV RMSE = {rmse:.2f}")

# ─── Step 6: Feature count explosion demonstration ───
print(f"\n=== Feature Explosion ===")
for p_feat in [1, 2, 5, 10]:
    for deg in [2, 3, 5]:
        poly = PolynomialFeatures(degree=deg)
        dummy = np.random.randn(5, p_feat)
        n_poly = poly.fit_transform(dummy).shape[1]
        print(f"  {p_feat} features × degree {deg} → {n_poly} polynomial terms")

# ─── Step 7: Visualization — Ridge tames high-degree polynomial ───
fig, axes = plt.subplots(1, 2, figsize=(14, 5))

for ax, (name, pipe) in zip(axes, [('Degree 10 — No Reg', pipe_no_reg),
                                     ('Degree 10 — Ridge', pipe_ridge)]):
    pipe.fit(months, revenue)
    y_plot = pipe.predict(months_plot)
    ax.scatter(months, revenue, alpha=0.5, s=20, color='#58a6ff')
    ax.plot(months_plot, y_plot, color='#f0883e', linewidth=2)
    ax.set_title(name)
    ax.set_xlabel('Months')
    ax.set_ylabel('Revenue')
    ax.set_ylim(revenue.min() - 200, revenue.max() + 200)
    ax.grid(True, alpha=0.2)

plt.suptitle('Regularization Tames High-Degree Polynomials', fontsize=13, color='#c9d1d9')
plt.tight_layout()
plt.savefig('project_20_ridge_polynomial.png', dpi=100, bbox_inches='tight')
print(f"✅ Ridge comparison saved as project_20_ridge_polynomial.png")

Key Takeaway

  • PolynomialFeatures transforms [x] into [x, x², x³, ...] — captures nonlinear patterns with a linear model
  • Low degree (2-3) usually suffices; high degree overfits catastrophically
  • Always scale before polynomial expansion — prevents numerical overflow
  • Combine with Ridge (Project 17) to use higher degrees safely
  • Use cross-validation (Project 15) to select the degree — never trust training error alone
  • Polynomial models are terrible at extrapolation — predictions outside training range diverge
