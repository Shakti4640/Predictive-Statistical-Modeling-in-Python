Project 56: Model Monitoring in Production — Detecting Data Drift

Data Drift Covariate Shift PSI Evidently AI Retraining Trigger

Difficulty: Very Hard  |  Builds on: Projects 52, 55

1 · Hypothetical Problem

  • A fintech company deployed a credit scoring model 6 months ago with 92% accuracy on the validation set
  • Over the past 3 months, the model's accuracy has silently dropped to 84% — but nobody noticed until customer complaints spiked
  • The root cause: economic conditions changed — interest rates rose, unemployment shifted, and the income distribution of applicants changed significantly
  • The model was trained on pre-change data, but the production input distribution has drifted — this is covariate shift
  • Goal → build a monitoring system that detects data drift in real time and triggers retraining alerts before accuracy degrades

2 · How to Approach It

  1. From Project 52: we learned to deploy models — but deployment without monitoring is a ticking time bomb
  2. From Project 55: we built pipelines — now we need to add monitoring as a pipeline stage
  3. Data drift means the distribution P(X) changes between training and production — even if P(Y|X) stays the same, predictions degrade
  4. We need statistical tests to compare training vs. production feature distributions
  5. Population Stability Index (PSI) is the industry standard for drift detection in credit scoring
  6. The Kolmogorov-Smirnov (KS) test provides a per-feature statistical test for distribution changes

3 · How Can We Solve It

  1. Simulate a credit scoring dataset with realistic feature distributions (income, age, debt ratio, credit history)
  2. Train a baseline model on the "original" distribution
  3. Simulate data drift by gradually shifting feature distributions over time (e.g., income drops, debt ratio increases)
  4. Implement Population Stability Index (PSI) from scratch to measure overall distribution shift
  5. Use the Kolmogorov-Smirnov test to detect per-feature drift with statistical significance
  6. Build a monitoring dashboard that tracks drift metrics over time and triggers retraining alerts when thresholds are exceeded
  7. Show how model accuracy correlates with drift magnitude

4 · What Should Do

  • ✅ Monitor ALL input features, not just model output — drift in inputs is the leading indicator
  • ✅ Set PSI thresholds: < 0.1 = no drift, 0.1–0.25 = moderate drift, > 0.25 = significant drift requiring action
  • ✅ Use multiple drift detection methods — PSI for overall shift, KS test for per-feature significance
  • ✅ Track drift over time with rolling windows — a single snapshot can be misleading
  • ✅ Automate retraining triggers — don't rely on humans to notice degradation
  • ✅ Monitor prediction distribution too — if the model suddenly predicts 90% "approve", something is wrong
  • ✅ Keep a reference dataset (training data summary) for comparison — you need a baseline

5 · What Should NOT Do

  • ❌ Do NOT deploy a model without monitoring — "deploy and forget" is the #1 cause of ML system failures
  • ❌ Do NOT only monitor accuracy — you often don't have ground truth labels in real time
  • ❌ Do NOT use a single metric — PSI alone misses feature-level drift; KS alone misses multivariate shifts
  • ❌ Do NOT set thresholds too sensitive — you'll get alert fatigue from false positives
  • ❌ Do NOT retrain on drifted data without understanding WHY it drifted — the drift might be a data pipeline bug, not a real-world change
  • ❌ Do NOT compare against stale reference data — update your reference periodically after confirmed retraining

6 · When to Use What

  • 🔹 PSI (Population Stability Index) → industry standard for credit scoring; good for continuous features binned into deciles
  • 🔹 KS Test (Kolmogorov-Smirnov) → per-feature statistical test; gives p-values for significance
  • 🔹 Jensen-Shannon Divergence → symmetric version of KL divergence; works for any distribution comparison
  • 🔹 Chi-Square Test → for categorical features; compares observed vs. expected frequencies
  • 🔹 Wasserstein Distance → "earth mover's distance"; captures the magnitude of shift, not just presence
  • 🔹 Multivariate Drift Detection → when features shift together; use domain classifier approach

7 · When NOT to Use

  • 🚫 Don't use PSI on very small samples (< 200) — the binning becomes unreliable
  • 🚫 Don't use KS test on categorical features — it assumes continuous distributions
  • 🚫 Don't rely solely on statistical tests when you have ground truth labels — direct accuracy monitoring is better
  • 🚫 Don't use drift detection as a substitute for proper data validation — garbage data isn't "drift"
  • 🚫 Don't apply the same thresholds across all features — some features naturally vary more than others

8 · How It Only Works — Internal Mechanics

Population Stability Index (PSI)

  • PSI measures how much a distribution has shifted from a reference (training) distribution
  • Step 1: Bin the reference distribution into N buckets (typically 10 deciles)
  • Step 2: Calculate the percentage of observations in each bucket for both reference and production
  • Step 3: PSI = Σ (P_production_i - P_reference_i) × ln(P_production_i / P_reference_i)
  • PSI is asymmetric — it measures how much production has shifted FROM the reference
  • Interpretation: PSI < 0.1 → stable; 0.1–0.25 → moderate shift; > 0.25 → significant shift

Kolmogorov-Smirnov Test

  • KS test compares two empirical CDFs and finds the maximum absolute difference
  • KS statistic = max|F_reference(x) - F_production(x)|
  • Returns a p-value — if p < 0.05, the distributions are statistically different
  • Advantage: non-parametric, no binning required, works on any continuous distribution
  • Limitation: sensitive to sample size — large samples detect tiny, irrelevant differences

Covariate Shift vs. Concept Drift

  • Covariate shift: P(X) changes but P(Y|X) stays the same — the input distribution shifts
  • Concept drift: P(Y|X) changes — the relationship between features and target changes
  • Both degrade model performance, but they require different responses
  • Covariate shift → retrain on new data distribution; Concept drift → may need a new model architecture

Monitoring Architecture

  • Reference window: a snapshot of the training data distribution (stored as histograms/statistics)
  • Production window: rolling window of recent production data (e.g., last 7 days)
  • Comparison: compute drift metrics between reference and production windows at regular intervals
  • Alerting: trigger retraining when drift exceeds thresholds for sustained periods

9 · Common Pitfalls and Failure Modes

  • ⚠️ Alert fatigue — setting thresholds too low triggers constant false alarms; teams start ignoring alerts
  • ⚠️ Confusing data bugs with drift — a broken ETL pipeline that sends nulls looks like drift but isn't
  • ⚠️ Ignoring multivariate drift — individual features may look stable, but their correlations change
  • ⚠️ Seasonal patterns misread as drift — holiday spending patterns aren't drift; they're expected variation
  • ⚠️ Stale reference data — if you never update the reference after retraining, you'll always detect "drift"
  • ⚠️ No ground truth delay accounting — in credit scoring, you don't know if a loan defaults for 6+ months

10 · Scalability Considerations

  • PSI computation is O(n) per feature — very fast even on millions of records
  • KS test is O(n log n) due to sorting — still fast for production use
  • Store reference distributions as histograms/quantiles, not raw data — reduces storage from GB to KB
  • Use streaming statistics (online mean, variance, quantiles) for real-time monitoring
  • For high-dimensional data, monitor the most important features first (use feature importance from the model)
  • Batch monitoring (daily/weekly) is sufficient for most use cases — real-time monitoring adds complexity

11 · Interpretability Considerations

  • ✅ PSI is intuitive — "the income distribution shifted by 0.3, which is significant"
  • ✅ Per-feature drift reports tell stakeholders exactly WHICH inputs changed
  • ✅ Drift visualizations (overlaid histograms) are easy for non-technical audiences to understand
  • ⚠️ Statistical test p-values can be misleading — "statistically significant" doesn't mean "practically important"
  • ⚠️ Multivariate drift is harder to explain — "the correlation between income and age changed" is abstract
  • ✅ Combine drift metrics with business metrics — "drift of 0.3 PSI correlated with 8% accuracy drop"

import numpy as np
import pandas as pd
from scipy import stats
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import accuracy_score, roc_auc_score
import matplotlib
matplotlib.use('Agg')
import matplotlib.pyplot as plt

np.random.seed(42)

# ═══════════════════════════════════════════════════════════════
# STEP 1: Generate Credit Scoring Dataset (Reference Period)
# ═══════════════════════════════════════════════════════════════
def generate_credit_data(n_samples, income_mean=50000, income_std=15000,
                         age_mean=40, debt_ratio_mean=0.3, drift_factor=0.0):
    """Generate synthetic credit scoring data with optional drift."""
    income = np.random.normal(income_mean * (1 - 0.2 * drift_factor),
                              income_std * (1 + 0.1 * drift_factor), n_samples)
    income = np.clip(income, 10000, 200000)

    age = np.random.normal(age_mean - 2 * drift_factor, 10, n_samples)
    age = np.clip(age, 18, 80).astype(int)

    debt_ratio = np.random.beta(2 + drift_factor, 5 - drift_factor * 0.5, n_samples)
    debt_ratio = np.clip(debt_ratio, 0, 1)

    credit_history = np.random.normal(650 - 20 * drift_factor, 80, n_samples)
    credit_history = np.clip(credit_history, 300, 850).astype(int)

    num_accounts = np.random.poisson(3 + drift_factor, n_samples)

    # Target: probability of default
    log_odds = (-3 + 0.00002 * (80000 - income) + 0.02 * (age - 40)
                + 3 * (debt_ratio - 0.3) + 0.005 * (650 - credit_history)
                + 0.1 * (num_accounts - 3))
    prob_default = 1 / (1 + np.exp(-log_odds))
    default = (np.random.random(n_samples) < prob_default).astype(int)

    return pd.DataFrame({
        'income': income, 'age': age, 'debt_ratio': debt_ratio,
        'credit_history': credit_history, 'num_accounts': num_accounts,
        'default': default
    })

# Reference data (training period)
df_reference = generate_credit_data(5000, drift_factor=0.0)
X_ref = df_reference.drop('default', axis=1)
y_ref = df_reference['default']

X_train, X_val, y_train, y_val = train_test_split(X_ref, y_ref, test_size=0.2, random_state=42)

# Train baseline model
model = GradientBoostingClassifier(n_estimators=100, max_depth=4, random_state=42)
model.fit(X_train, y_train)
baseline_acc = accuracy_score(y_val, model.predict(X_val))
baseline_auc = roc_auc_score(y_val, model.predict_proba(X_val)[:, 1])
print(f"=== Baseline Model Performance ===")
print(f"  Validation Accuracy: {baseline_acc:.4f}")
print(f"  Validation AUC: {baseline_auc:.4f}")

# ═══════════════════════════════════════════════════════════════
# STEP 2: Implement PSI From Scratch
# ═══════════════════════════════════════════════════════════════
def calculate_psi(reference, production, n_bins=10):
    """Calculate Population Stability Index between two distributions."""
    # Create bins from reference distribution
    breakpoints = np.percentile(reference, np.linspace(0, 100, n_bins + 1))
    breakpoints[0] = -np.inf
    breakpoints[-1] = np.inf
    # Remove duplicate breakpoints
    breakpoints = np.unique(breakpoints)

    # Calculate proportions in each bin
    ref_counts = np.histogram(reference, bins=breakpoints)[0]
    prod_counts = np.histogram(production, bins=breakpoints)[0]

    # Convert to proportions (add small epsilon to avoid division by zero)
    eps = 1e-4
    ref_pct = ref_counts / len(reference) + eps
    prod_pct = prod_counts / len(production) + eps

    # Normalize to sum to 1
    ref_pct = ref_pct / ref_pct.sum()
    prod_pct = prod_pct / prod_pct.sum()

    # PSI formula
    psi = np.sum((prod_pct - ref_pct) * np.log(prod_pct / ref_pct))
    return psi

print(f"\n=== PSI Implementation Test ===")
# Test with identical distributions
same_data = np.random.normal(0, 1, 5000)
psi_same = calculate_psi(same_data, np.random.normal(0, 1, 5000))
print(f"  PSI (same distribution): {psi_same:.4f} — should be near 0")

# Test with shifted distribution
psi_shifted = calculate_psi(same_data, np.random.normal(0.5, 1.2, 5000))
print(f"  PSI (shifted distribution): {psi_shifted:.4f} — should be > 0.1")

# ═══════════════════════════════════════════════════════════════
# STEP 3: Simulate Drift Over Time (6 Monthly Windows)
# ═══════════════════════════════════════════════════════════════
drift_factors = [0.0, 0.2, 0.5, 1.0, 1.5, 2.0]
months = ['Month 1', 'Month 2', 'Month 3', 'Month 4', 'Month 5', 'Month 6']
features = ['income', 'age', 'debt_ratio', 'credit_history', 'num_accounts']

results = []
print(f"\n=== Drift Simulation Over 6 Months ===")
print(f"  {'Month':<10} {'Accuracy':>10} {'AUC':>8} {'PSI_income':>12} {'PSI_debt':>10} {'Alert':>8}")
print(f"  {'-'*58}")

for i, drift in enumerate(drift_factors):
    df_prod = generate_credit_data(1000, drift_factor=drift)
    X_prod = df_prod.drop('default', axis=1)
    y_prod = df_prod['default']

    # Model performance on drifted data
    acc = accuracy_score(y_prod, model.predict(X_prod))
    auc = roc_auc_score(y_prod, model.predict_proba(X_prod)[:, 1])

    # Calculate PSI for each feature
    psi_values = {}
    for feat in features:
        psi_values[feat] = calculate_psi(X_ref[feat].values, X_prod[feat].values)

    # KS test for each feature
    ks_results = {}
    for feat in features:
        ks_stat, ks_pval = stats.ks_2samp(X_ref[feat].values, X_prod[feat].values)
        ks_results[feat] = {'statistic': ks_stat, 'p_value': ks_pval}

    # Alert logic
    max_psi = max(psi_values.values())
    alert = "🔴 ALERT" if max_psi > 0.25 else ("🟡 WARN" if max_psi > 0.1 else "🟢 OK")

    results.append({
        'month': months[i], 'drift_factor': drift, 'accuracy': acc, 'auc': auc,
        'psi': psi_values, 'ks': ks_results, 'alert': alert, 'max_psi': max_psi
    })

    print(f"  {months[i]:<10} {acc:>10.4f} {auc:>8.4f} {psi_values['income']:>12.4f} "
          f"{psi_values['debt_ratio']:>10.4f} {alert:>8}")

# ═══════════════════════════════════════════════════════════════
# STEP 4: Detailed KS Test Results for Most Drifted Month
# ═══════════════════════════════════════════════════════════════
print(f"\n=== KS Test Results — Month 6 (Heaviest Drift) ===")
last = results[-1]
print(f"  {'Feature':<18} {'KS Stat':>10} {'p-value':>12} {'Significant':>12}")
print(f"  {'-'*54}")
for feat in features:
    ks = last['ks'][feat]
    sig = "YES ⚠️" if ks['p_value'] < 0.05 else "no"
    print(f"  {feat:<18} {ks['statistic']:>10.4f} {ks['p_value']:>12.6f} {sig:>12}")

# ═══════════════════════════════════════════════════════════════
# STEP 5: Retraining Trigger Logic
# ═══════════════════════════════════════════════════════════════
print(f"\n=== Retraining Trigger Analysis ===")
psi_threshold = 0.25
consecutive_alerts = 0
trigger_month = None

for r in results:
    if r['max_psi'] > psi_threshold:
        consecutive_alerts += 1
        if consecutive_alerts >= 2 and trigger_month is None:
            trigger_month = r['month']
    else:
        consecutive_alerts = 0

if trigger_month:
    print(f"  ⚡ Retraining triggered at: {trigger_month}")
    print(f"  Reason: PSI exceeded {psi_threshold} for 2 consecutive months")
else:
    print(f"  ✅ No retraining needed — drift within acceptable bounds")

# Show accuracy recovery after retraining
df_retrain = generate_credit_data(5000, drift_factor=1.5)
X_retrain = df_retrain.drop('default', axis=1)
y_retrain = df_retrain['default']
model_retrained = GradientBoostingClassifier(n_estimators=100, max_depth=4, random_state=42)
model_retrained.fit(X_retrain, y_retrain)

df_test_new = generate_credit_data(1000, drift_factor=2.0)
X_test_new = df_test_new.drop('default', axis=1)
y_test_new = df_test_new['default']

acc_old = accuracy_score(y_test_new, model.predict(X_test_new))
acc_new = accuracy_score(y_test_new, model_retrained.predict(X_test_new))
print(f"\n  Performance on latest data:")
print(f"    Old model accuracy: {acc_old:.4f}")
print(f"    Retrained model accuracy: {acc_new:.4f}")
print(f"    Improvement: +{acc_new - acc_old:.4f}")
